# 🧱 如何让 AI 写出靠谱的项目规则

> 把「写 README」这种高耗能工作，交给 AI，但别让它乱写。

很多人第一次让 AI 写项目规则，往往是这样的流程：

- 打开聊天框
- 敲一句：**“帮我写一份项目约定”**
- 回车
- ……然后拿到一份**放到哪个项目里都不会出事**的通用模板 🥲

问题不在 AI 不行，而在于 prompt 太随意。

这篇小文就做一件事：

> 教你一个**可复用模板**，  
> 让 AI 主动「扫仓库 + 总结约定 + 写成规则」，  
> 而不是瞎编一堆「放之四海而皆准」的废话。

---

## 🎯 目标：写给“未来的 AI 同事”看的规则

普通 README 是写给人看的，而这里的 project rules，有一个更具体的读者：

> **之后在这个仓库里帮你写代码的 AI。**

所以你希望文档回答的不是「我们用 React」，而是：

- 新页面**该放哪**？
- 公共组件**长什么样**？
- 状态是用啥管的（Redux？Zustand？Jotai？都不是？）
- 内容是写死在代码里，还是走 Markdown / CMS / XLSX？
- 想做一个改动，**应该动哪些文件，最好别动哪些？**

只要 AI 知道这些，它就会更像一个「懂项目脾气」的同事。

---

## 🧩 Prompt 的三块拼图

让 AI 写这种项目规则，prompt 里只要讲清三件事：

- **这是个什么项目**  
  比如：Next.js 前端 + Netlify 函数 + Tailwind + Markdown 内容。

- **你允许它干到什么程度**  
  例如：可以自由浏览仓库文件，但不要瞎改代码，只做“阅读 +总结”。

- **你想要的文档长什么样**  
  例如要哪些章节、面向 AI 写、重点说明“新增代码/文件应该放哪”。

只要把这三块信息塞进 prompt，AI 基本就不会跑偏。

## 🧱 可复用 Prompt 模板（版本一）

下面是一个可以直接复用的模板，你只需要改一两句描述，就能在任何仓库里用：

```text
请你扮演一个“项目上下文架构师 + AI Agent 规范设计师”。
当前这是一个前端代码仓库（例如：Next.js + React + Tailwind + 一些后端函数）。

你需要做两件事：
1. 快速分析这个仓库的实际情况：
   - 技术栈（框架、语言、主要依赖）
   - 目录结构与分层（页面、组件、工具、内容、配置）
   - 路由和状态管理方案
   - 样式体系（CSS / Tailwind / UI 组件库等）
   - 构建与部署方式
   你可以自行浏览 package.json、src 目录、配置文件（如 next.config.*、tailwind.config.*）和任何你认为重要的文件。

2. 基于分析结果，输出一份“面向 AI Agent 的项目规则文档（project rules）”，
   用于指导之后在本仓库中运行的 AI 应该：
   - 写代码时遵守哪些约定
   - 新增页面 / 组件 / 工具函数 / 内容文件时应该放在哪里
   - 优先复用哪些已有能力，而不是重复造轮子

文档请用「中文 + Markdown」输出，结构参考（可按实际增删）：
- Frontmatter（如果是 Cursor 规则文件，可包含 description、alwaysApply 等字段）
- ## 项目约定
  - 代码风格、命名规范、目录与文件组织方式
- ## 架构模式
  - 核心目录说明
  - 路由与导航约定
  - 状态管理方式
- ## 内容与数据驱动机制（如果有）
  - Markdown / CMS / XLSX / API 等的分工
  - 哪些数据是配置驱动，哪些是代码写死
- ## 构建与部署约定
  - 本地开发命令
  - 构建方式
  - 部署平台与主要配置
- ## 外部依赖与约束
  - 关键依赖版本（框架、UI 库、状态管理库等）
  - 不建议随意变动的前提（比如必须使用某个部署平台或 UI 库）
- ## AI Agent 使用指引（重点）
  - 不同类型的改动（页面、组件、样式、内容、脚本），应该改哪些文件
  - 新功能时，优先复用哪些现有组件 / Hook / 工具方法
  - 在哪些情况下可以引入新依赖，以及需要在文档中补充什么

要求：
- 只能基于当前仓库真实存在的代码和配置，不要胡编通用规则。
- 尽量用小节和列表，保持简洁但信息密度高。
- 特别强调“新增代码/文件应该放到哪里”和“应优先复用哪些现有能力”。
```

## 🛠 小技巧：怎么用更爽？

- **第一次用**：让 AI 跑一遍上面的模板，生成一份「项目规则初稿」。
- **觉得不错时**：把这份文档丢进 `.cursor/rules/`，变成真正的「项目内置规则」。
- **项目大升级时**：再跑一次，让 AI 更新规则，顺便当一遍架构回顾。

你甚至可以对 AI 说：

> 「以后在这个仓库里写代码，先按这份规则来，不清楚的地方再问我。」

从这一刻起，它就不是一个“通用聊天机器人”，  
而是一个**读过项目手册的新同事**了。🚀
