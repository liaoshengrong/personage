## 通用 Prompt 模板（适用于任何前端项目）

> 用于让 AI 自动「读仓库 → 提炼现状 → 生成项目级 Coding Rules 文档」，可以在任意前端项目里复用。

```text
请你扮演「前端团队的代码规范设计师 + 架构师」。
当前这是一个前端项目仓库（包含 React / Vue / Next.js / 构建配置等代码），我希望你：
1. **先阅读并分析本仓库**：
   - 查看 `package.json`、`tsconfig.json`、`eslint`/`prettier` 配置
   - 浏览 `src` 目录结构，了解组件、页面、hooks、utils、types 等的组织方式
   - 注意现有的命名风格、导入顺序、状态管理、异步处理、样式方案等
2. **基于真实代码，总结一份“项目级前端 Coding Rules 文档”**，而不是凭空想象。

文档请使用 **中文 + Markdown** 输出，风格专业但口语一点，结构参考：
- ## 总览  
  - 简要说明本项目的技术栈、主要框架、构建/部署方式
- ## 代码风格（Code Style）  
  - 语言选择（TS/JS）、函数式组件、hooks 使用约定  
  - 命名规则（文件、组件、变量、hook、常量、枚举）  
  - 导入顺序、相对/绝对路径约定  
  - eslint / prettier / editorconfig 相关规范  
- ## 目录与模块划分（Structure）  
  - 目录结构说明，每一层的职责  
  - 页面 / 组件 / hooks / utils / types 分放在哪  
- ## 组件与状态（Components & State）  
  - 函数组件写法、props 设计原则  
  - 状态管理方案（本地 state / 全局状态库 / URL / params）  
  - 副作用处理（useEffect/useQuery 等的约定）  
- ## 样式与 UI（Styling）  
  - Tailwind / CSS Modules / UI 库 的使用方式和禁用行为  
  - 响应式、主题、布局相关约定  
- ## API 与异步（API & Async）  
  - 接口封装位置与调用方式  
  - 错误处理、loading 状态、重试策略等约定  
- ## 测试与质量（Testing & Quality）  
  - 单测 / e2e 大致策略（如果项目里已有，可以总结；没有就轻描淡写）  
- ## 典型场景怎么写（Recipes）  
  - 新增页面 / 新增组件 / 新增接口 / 新增文章时，推荐的落点和写法

要求：
- 所有规则尽量**参考当前仓库已有实践**，不要写那种「放哪都适用」的废话。
- 可以适度给出 **“❌ 错误示例 / ✅ 推荐写法”**，但要简短。
- 重点回答「**以后在这个项目写代码，应该怎么统一写？**」，而不是讲通用前端知识。
```
